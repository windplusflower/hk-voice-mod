## 1. Requirements (Context)
- **Goal**: 研究并实现把 `HkVoiceMod` 菜单统一改成“自定义宏”配置体系：现有固定命令转为预设宏，用户可以编辑宏步骤、延迟、唤醒词与阈值；`Stop` 作为特殊项单独保留，仅允许配置唤醒词与阈值。
- **In-Scope**:
  - 梳理当前菜单、设置、命令定义、命令执行、输入注入的真实实现位置
  - 判断“当前项目到底支持什么程度的自定义”
  - 识别哪些现有命令可直接转成“预设宏”
  - 识别宏系统引入后必然要改动的数据模型与执行链路
  - 评估 `Satchel.BetterMenus` 是否足以承载宏编辑页 + 按键捕获 + 延迟编辑交互
- **Out-of-Scope**:
  - 本轮不改代码
  - 本轮不直接敲定最终 UI 视觉细节
  - 本轮不做持久化结构重构

## 1.5 Code Map (Project Topology)
- **Menu / UI**:
  - `HkVoiceMod/HkVoiceMod.cs`: Mod 入口，已实现 `ICustomMenuMod`，`GetMenuScreen(...)` 直接交给 `VoiceSettingsMenuBuilder`。
  - `HkVoiceMod/Menu/VoiceSettingsMenuBuilder.cs`: 当前唯一菜单构建器；使用 `Satchel.BetterMenus.Menu` 线性追加 `TextPanel + InputField + FloatInputField + MenuButton`，现在是单页垂直表单，还没有“首页 -> 子页面”的菜单分层。
  - `Satchel.BetterMenus` 外部资料确认：
    - `Menu` 默认构建的是**单个垂直滚动内容区**，不提供原生双面板 API；
    - `MenuRow` 支持“同一行多个元素横向摆放”，但这更适合一行双按钮，不适合复杂双编辑器面板；
    - 若要做宏录制器这类复杂交互，BetterMenus 需要走 `StaticPanel` 或自定义 `Element` 扩展，而不是依赖现成控件。
  - `HkVoiceMod/Menu/VoiceSettingsDraft.cs`: 菜单 draft 容器；菜单只编辑 `VoiceModSettings` 的临时副本，点击 `Apply` 才提交。
- **Settings / Persistence**:
  - `HkVoiceMod/VoiceModSettings.cs`: 当前持久化模型。真正可配置的是 `List<VoiceCommandKeywordConfig> CommandKeywordConfigs`，即 11 个固定命令的“唤醒词 + 阈值”。没有“自定义命令列表”或“宏列表”。
  - `HkVoiceMod/Commands/VoiceCommandKeywordConfig.cs`: 单条命令配置对象，只包含 `Command / WakeWord / KeywordThreshold`。
  - 按最新需求，后续设置层需要从“仅关键词配置”扩展为：“可增删的宏定义列表 + 唤醒词阈值绑定 + 顶部固定的 Stop 特殊配置”。
- **Command Model**:
  - `HkVoiceMod/Commands/VoiceCommand.cs`: 固定 11 个枚举值：`Up / Down / Left / Right / Attack / Jump / Dash / Howl / Dive / Cast / Stop`。
  - `HkVoiceMod/Commands/VoiceCommandCatalog.cs`: 11 条固定命令目录，包含显示名、默认唤醒词、默认阈值；菜单就是按这里的固定顺序渲染。
  - `HkVoiceMod/Commands/VoiceCommandMap.cs`: `VoiceCommand -> KeyActionProfile` 映射；这里定义了每个固定命令最终触发的键动作。
  - `HkVoiceMod/Commands/KeyActionProfile.cs`: 描述单个命令的执行语义：`Mode / Keys / DurationSeconds / ReleaseOppositeHorizontalHold`。
  - `HkVoiceMod/Commands/HeroActionKey.cs`: 当前可注入的底层动作键只有 `Left / Right / Up / Down / Attack / Jump / Dash / Cast`。
- **Runtime / Execution**:
  - `HkVoiceMod/Runtime/VoiceRuntimeController.cs`: 接收识别结果并调用 `_inputInjector.Dispatch(command, now)`；现阶段只会分发固定 `VoiceCommand`。
  - `HkVoiceMod/Input/HeroActionInputInjector.cs`: 真正执行按键注入。内部只理解 `KeyActionProfile`，支持 `Tap / TimedHold / ContinuousHold / ReleaseContinuous`，没有“步骤序列宏”概念。
- **Recognition / Apply Flow**:
  - `HkVoiceMod/HkVoiceMod.cs::TryApplyVoiceCommandSettings(...)`: 菜单 Apply 后会校验/编译/更新 `Settings` 并重启后端。
  - `HkVoiceMod/Recognition/Sherpa/SherpaKeywordSpotterBackend.cs`: 识别后通过 `_commandLookup` 把关键词映射回固定 `VoiceCommand`；识别结果输出仍然是单个固定命令，不是任意宏 ID。

## 2. Architecture (Optional - Populated in INNOVATE)
- **已确认事实**:
  - 当前项目并没有“真正的自定义指令系统”；现有菜单只是“11 个固定操作的唤醒词与阈值编辑器”。
  - 用户最新决策进一步收敛了模型：**不再维护“自定义指令”与“自定义宏”两套概念**，而是统一为“宏”，其中单动作命令只是单步骤宏。
  - 当前执行链路是：`关键词 -> VoiceCommand -> KeyActionProfile -> HeroActionInputInjector`，其中 `VoiceCommand` 是中心主键。
  - 以“统一宏”视角看，除 `Stop` 外，现有命令都可视作预设宏：
    - `Howl` -> 双键同发宏 `Up + Cast`
    - `Dive` -> 双键同发宏 `Down + Cast`
    - `Up/Down/Left/Right/Attack/Jump/Dash/Cast` -> 单步骤宏
    - `Cast` 的展示与默认唤醒词都应改为 `法术`
    - `Stop` 保持特殊控制命令，不进入宏编辑器
  - 用户新增确认了 CRUD 规则：
    - `Stop` 固定在列表最上方；
    - `Stop` 不允许删除；
    - 其他宏允许添加；
    - 其他宏允许删除；
    - 首版允许把除 `Stop` 外的其余预设宏全部删空。
  - 现有 `KeyActionProfile` 只表达“一次命令映射为一组同时生效的键 + 一个持续时间/模式”，无法表达 `S J 0.5 W` 这种分步骤宏。
  - 用户已明确要求宏编辑展示名遵循**游戏当前按键设置**，因此宏系统不能直接把 `S/J/W` 当固定 DSL 字母，而必须先把键盘输入映射为“当前绑定到某个 HeroAction 的游戏键名”；映射不到的键要丢弃。
  - Librarian 外部调研确认：BetterMenus 没有现成的宏录制控件；宏编辑器若要支持按键捕获、延迟插入、特殊键处理，基本要自定义 `Element` 或在 `StaticPanel` 里挂自绘 UI。
  - 如果严格按“统一宏”建模，识别层长期最好从“关键词 -> VoiceCommand”演进为“关键词 -> MacroDefinitionId / ActionDefinitionId”；但 Oracle 结论指出，这属于高改动路线。
- **初步架构判断**:
  - 这不只是 UI 改造；核心变化是要把“固定动作映射”升级成“可编辑宏定义”。
  - 结合 Oracle 结论，最低风险路线不是立刻让识别层脱离 `VoiceCommand`，而是：
    - 暂时保留 `VoiceCommand` 作为识别与绑定主键；
    - 把除 `Stop` 外的固定命令全部视为“内置预设宏槽位”；
    - 每个槽位都可编辑宏体、唤醒词、阈值；
    - `Stop` 保留独立配置与独立执行语义。
  - 更干净的长期方向仍然是把“语音唤醒词配置”和“动作执行定义”拆层：
    1. 一层定义可识别命令项（显示名、唤醒词、阈值、目标动作 ID）；
    2. 一层定义动作实现（固定单动作或宏步骤序列）；
    3. 运行时将识别结果派发给统一的动作执行器，而不是写死 `VoiceCommandMap`。
  - 按当前最新需求做最小化落地，过渡方案应改成：
    - 不再做“指令页/宏页”二分；
    - 菜单统一呈现“宏列表”，列表顶部固定 `Stop` 特殊项，其余列表项是可增删的宏；
    - `Howl/Dive` 从硬编码 `VoiceCommandMap` 迁移为默认宏；
    - `Up/Down/Left/Right/Attack/Jump/Dash/Cast` 也统一以单步骤宏形式呈现；
    - `Stop` 单独作为特殊配置项，不允许进入宏编辑器，也不允许删除；
    - 其余宏支持新增与删除；
    - 其余预设宏不要求保底数量，允许全部删除；
    - 新增宏编辑器先支持“动作键步骤 + 延迟步骤”；
    - 识别层仍然只认固定 `VoiceCommand`，先不扩展成无限自定义语音命令。
  - Oracle 进一步给出的低风险执行建议是：
    1. 先不要让识别层脱离 `VoiceCommand`；
    2. 在 `VoiceRuntimeController.DrainRecognizedCommands()` 与 `_inputInjector.Dispatch(...)` 之间插入一层“宏执行器/动作解析器”；
    3. `HeroActionInputInjector` 保持“底层按键写入器”角色，宏的时间调度放在其上层；
    4. `Howl / Dive` 先作为默认宏迁移，随后把其余普通命令也收敛到同一“预设宏”模型；
    5. `Stop` 继续走独立分支，不纳入宏调度器。

## 3. Detailed Design & Implementation (Populated in PLAN)

### 3.1 总体实现策略

- 本次实现采用 **“动态宏 ID + 固定 Stop 特殊项”** 的正式模型，不再继续以 `VoiceCommand` 作为运行时主键。
- `Stop` 不是宏，也不参与增删；它是固定触发项，永远显示在菜单最上方，只配置：
  - `WakeWord`
  - `KeywordThreshold`
- 其余条目全部进入 `MacroConfigs`：
  - 可新增
  - 可删除
  - 可编辑显示名、唤醒词、阈值、步骤序列
  - 首版允许删空
- 宏步骤首版只支持两种：
  1. `Action`：一个动作步骤，可包含 1 个或多个 `HeroActionKey`
  2. `Delay`：延迟若干秒
- 动作步骤不直接让用户编辑 `KeyPressMode`；其执行语义按当前项目内置动作默认值生成：
  - `Left / Right` -> `ContinuousHold`
  - `Up / Down / Jump` -> `TimedHold`
  - `Attack / Dash / Cast` -> `Tap`
  - 多键动作（如 `Up + Cast`）使用同一 `Action` 步骤同时触发

### 3.3 数据结构与接口定义

- `StopKeywordConfig`
  - 固定只有一条，不允许删除
  - 默认值：
    - `WakeWord = "停止"`
    - `KeywordThreshold = 0.1f`

- `VoiceMacroConfig`
  - `Id`：稳定唯一 ID，保存到设置文件，用于识别结果和菜单编辑定位
  - `DisplayName`：菜单显示名，可编辑
  - `WakeWord`：宏的唤醒词
  - `KeywordThreshold`：宏的阈值
  - `Steps`：按顺序执行的步骤列表
  - `IsPreset`：仅作 UI 提示；不影响可删性

- `VoiceMacroStep`
  - 当 `StepKind == Action`：
    - `Keys.Count >= 1`
    - `PressMode`、`DurationSeconds`、`ReleaseOppositeHorizontalHold` 有效
    - `DelaySeconds == 0`
  - 当 `StepKind == Delay`：
    - `DelaySeconds > 0`
    - `Keys.Count == 0`

- `RecognizedTriggerEvent`
  - `TriggerKind == Stop` 时：
    - `TriggerId = "stop"`
  - `TriggerKind == Macro` 时：
    - `TriggerId = VoiceMacroConfig.Id`

### 3.4 菜单与交互蓝图

- 统一菜单只保留一套入口：`自定义宏`
- 打开后页面顺序固定为：
  1. `Stop` 配置区（最上方）
  2. 宏列表
  3. `Add Macro`
  4. `Apply`

- `Stop` 配置区：
  - 输入项：
    - 唤醒词
    - 阈值
  - 禁止显示：
    - 删除按钮
    - 步骤编辑器

- 宏列表区：
  - 每条宏显示：
    - `DisplayName`
    - `WakeWord`
    - `KeywordThreshold`
    - 步骤摘要文本
    - `Edit`
    - `Delete`

- `Add Macro`：
  - 追加一个新的空宏草稿，默认值：
    - `Id = Guid/NewId`
    - `DisplayName = "新宏"`
    - `WakeWord = ""`
    - `KeywordThreshold = 0.1f`
    - `Steps = []`

- 宏编辑器页（由普通 BetterMenus 控件 + `VoiceMacroCaptureService` 驱动）：
  - 可编辑字段：
    - `DisplayName`
    - `WakeWord`
    - `KeywordThreshold`
  - 步骤区能力：
    - 聚焦录制区域后，玩家按下键盘键
    - 若当前键能映射到受支持 `HeroActionKey`，则追加一个 `Action` 步骤
    - 若当前键无法映射到受支持动作，则忽略
    - 点击 `Delay` 按钮后，再输入时间，追加一个 `Delay` 步骤
    - `Backspace`：删除末尾步骤
    - `Esc`：取消本次编辑页修改并返回
    - `Enter`：确认本次编辑页修改并返回
  - 摘要显示格式：
    - 例如 `跳跃 法术 0.5 上移`
    - 延迟显示为数字秒值

### 3.5 运行时与迁移流程

- `OnLoadGlobal()` / `Apply` 的统一处理顺序：
  1. `Clone()` 当前设置
  2. `MigrateLegacyCommandConfigsIfNeeded()`
  3. `EnsureMacroDefaults()`
  4. `NormalizeAndValidateMacroSettings()`
  5. 调用 `SherpaKeywordCompiler.Compile(modelDirectory, stopConfig, macroConfigs)`
  6. 若成功：提交 `Settings`、刷新 `VoiceMacroRunner`、重启识别后端
  7. 若失败：保留旧设置和旧运行时状态

- 旧设置迁移规则：
  - 若发现：
    - `MacroConfigs` 为空或缺失
    - `CommandKeywordConfigs` 有旧数据
  - 则执行迁移：
    - `Stop` 从旧 `VoiceCommand.Stop` 生成 `StopKeywordConfig`
    - 其余旧命令逐条生成 `VoiceMacroConfig`
    - `Cast` 迁移后显示名与默认唤醒词改为 `法术`
    - `Howl` 迁移为一个双键动作步骤宏
    - `Dive` 迁移为一个双键动作步骤宏

- Stop 运行时语义：
  - 调用 `VoiceMacroRunner.CancelPendingSteps()`
  - 调用 `HeroActionInputInjector.ReleaseContinuousInputs()`
  - 不清除已触发且正在倒计时的 timed keys

### 3.6 校验规则

- 全局唯一性：
  - `StopKeywordConfig.WakeWord`
  - 所有 `MacroConfigs[*].WakeWord`
  - 三者合并后必须全局唯一

- `StopKeywordConfig`：
  - `WakeWord` 非空
  - `KeywordThreshold` 在 `0.01 ~ 1.00`

- `VoiceMacroConfig`：
  - `Id` 非空且唯一
  - `DisplayName` 非空
  - `WakeWord` 非空且规范化后唯一
  - `KeywordThreshold` 在 `0.01 ~ 1.00`
  - `Steps.Count >= 1`

- `VoiceMacroStep`：
  - `Delay` 步骤必须 `DelaySeconds > 0`
  - `Action` 步骤必须至少有一个 `HeroActionKey`
  - 首版只允许受支持的 8 个 `HeroActionKey`

- 唤醒词字符规则：
  - 继续沿用当前 `NormalizeWakeWord(...)` 与中文字符限制
  - 任何违反当前 Sherpa 编译约束的文本都拒绝 Apply
