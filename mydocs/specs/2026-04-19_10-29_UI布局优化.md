## 2. Architecture (Optional - Populated in INNOVATE)
- **当前已确认的布局事实**:
  1. 主窗口尺寸固定：`WindowWidth = 1460f`，窗口内容区由 `VerticalLayoutGroup` 管理。
  2. Stop 区使用两个 `CreateLabeledInput(...)` 生成的“二级横向组”并排放入 `stopRow`；但组本身没有显式首选宽度，而父横向布局又关闭了宽度控制与扩展，这会导致子组宽度分配不稳定，是上方控件互相挤压/重叠的高风险点。
  3. 宏第一行使用固定宽度常量：
     - `MacroWakeWordWidth = 460f`
     - `MacroThresholdWidth = 180f`
     - `SecondaryButtonWidth = 140f`
     - `DeleteButtonWidth = 120f`
     - 标签 `宏 N` 还固定了 `90f`
     这会让整行只占用一部分可用宽度，剩余空间闲置在右侧，形成截图中的明显留白。
  4. 宏第二行摘要文本使用固定高度 `MacroSummaryHeight = 52f`，同时 `verticalOverflow = Truncate`；这意味着长摘要会被硬截断，而不是自然增高。虽然这不是本次用户点名的问题，但它和整体排版紧凑感直接相关。
  5. Footer 按钮区同样依赖固定按钮宽度，文本默认允许换行；像 `Discard` 这类较长/较大字号文本，已经出现断行迹象，说明按钮文本没有被强制单行保护。
  6. 当前工程内没有 `GridLayoutGroup` 或“自动按列换行”的通用 helper；现有布局全部基于 `VerticalLayoutGroup + HorizontalLayoutGroup + LayoutElement` 组合，因此“每行四项自动铺满”的能力并不存在，需要显式改造。
- **本轮问题归因（研究结论）**:
  - **问题 A：每行四个选项没有占满一行**
    - 根因不是 ScrollView 宽度不够，而是宏第一行仍采用“多个固定宽度子项并列”的策略，没有为核心输入项提供弹性宽度，也没有为整行做按剩余空间分配。
    - 最直观可调点在 `BuildMacroRow(...)` 与 `CreateHorizontalLayout(...)`。
    - 代理复核后的尺寸证据：宏行内容区可用宽度约为 `1320px`，而当前 top row 固定控件宽度合计仅约 `990px`，加上 4 处间距后约 `1038px`，仍会剩下约 `282px` 空白，因此截图中的右侧留白是当前实现的必然结果。
  - **问题 B：上方按钮/控件重叠**
    - 高概率根因在 `StopSection` 的 `CreateLabeledInput(...)` 结构：父级 `stopRow` 是横向组，子级每个 labeled group 也是横向组，但该子组没有明确首选宽度，且父布局不控制子宽度，最终造成两个复合控件并排时的宽度压缩与内容串位。
    - 这比单纯调小字体更像是容器约束错误。
    - 代理复核后的高度证据：`Header` 当前固定高 `124f` 低于其三行文本 + padding + spacing 的实际需求；`StopSection` 固定高 `132f` 也低于其标题 + 行容器 + padding + spacing 的真实占用。因此“上方重叠”除了横向宽度约束外，还包含 section 预设高度不足的问题。
- **下一阶段设计方向（供 PLAN 使用）**:
  1. Stop 区不要继续把两个“标签+输入框”复合组直接裸并排；要么给每个 group 明确总宽度，要么改成单层横向布局并精确控制 label/input 的占比。
  2. 宏第一行应把“唤醒词输入框”作为弹性主列，阈值与按钮维持较小固定宽度，从而吃掉右侧空白。
  3. 需要给按钮文本增加“单行展示”约束，避免 `Discard` 这类按钮在窄宽度下折行。
  4. 若继续保留摘要区双行结构，摘要文本高度应改为可自适应或至少提高，以减少压缩感。

### 3.2 Layout Strategy

- **总体原则**: 采用“局部结构修正 + 局部弹性列”方案，不做大规模 UI helper 重写。

- **Stop 区策略**:
  - 目标：修复当前顶部复合输入组挤压/串位/重叠风险。
  - 方案：
    1. 不再依赖两个 `CreateLabeledInput(...)` 复合横向组直接并排。
    2. 在 `StopSection` 内构建一个单层横向布局：
       - `Stop 唤醒词 Label`
       - `Stop 唤醒词 Input`
       - `Stop 阈值 Label`
       - `Stop 阈值 Input`
    3. 标签使用固定窄宽度；输入框分别使用较宽固定宽度或“保底宽度 + 弹性宽度”。
    4. `StopSection` 自身高度上调，确保标题 + row + padding + spacing 不再溢出。

- **Macro 第一行策略**:
  - 目标：让用户感知上的 4 个主选项（唤醒词 / 阈值 / 录制 / 删除）尽可能占满一行。
  - 方案：
    1. 保留 `宏 N` 标签，但将其视为窄辅助列，而不是主内容列。
    2. `WakeWord Input` 改为主弹性列。
    3. `Threshold Input` 保持固定窄列。
    4. `Record` / `Delete` 保持固定按钮宽度。
    5. 若需要填满剩余空间，优先把额外宽度分配给 `WakeWord Input`，而不是简单插入右侧 spacer。
  - 结果预期：
    - 第一行视觉上不再左挤右空。
    - 用户可编辑的主输入项会更宽。
    - 不破坏按钮宽度的稳定点击区域。

- **Header / Footer 策略**:
  - Header：提高 section 高度，保证标题、副标题、状态文本三行稳定展示。
  - Footer：保持现有 `CreateSpacer(...)` 的右对齐策略，但增大 `Discard` 宽度，并给按钮文本加单行约束。

- **摘要行策略**:
  - 本轮主要问题不是摘要区，但为了避免布局观感继续紧绷：
    - 维持双行结构不变；
    - 暂不在本轮把摘要改成完全自适应高度；
    - 如果修改第一行宽度后仍显得拥挤，再作为后续独立优化项处理。
