# SDD Spec: 宏录制按键映射来源

## 0. 🚨 Open Questions (MUST BE CLEAR BEFORE CODING)
- None
- 已确认目标：扩展到截图中的**游戏键位页动作集合**，且录制侧**不硬编码键位映射**，必须跟随游戏当前键位设置自适应。
- 设计假设：录制支持范围以截图对应的**游戏内可绑定玩法动作**为准，默认**不包含** `MENU_* / pause / textSpeedup / skipCutscene / rs_* / focus / evade` 等非截图主玩法项。

## 1. Requirements (Context)
- **Goal**: 让宏录制与宏回放对齐 Hollow Knight 键位设置页：录制时按物理键会映射到当前游戏绑定的动作；玩家改键后，无需改 mod 配置，录制侧自动适配。
- **In-Scope**:
  - 录制支持截图对应的玩法动作：`UP/DOWN/LEFT/RIGHT/JUMP/ATTACK/DASH/SUPER_DASH/CAST/QUICK_CAST/DREAM_NAIL/QUICK_MAP/INVENTORY`
  - 显示层继续展示动作名，但其来源改为游戏原生动作/本地化入口
  - 回放层能对上述动作做注入
  - 录制弹窗新增显式按钮：`开始录制 / 停止录制 / 清空`
  - 进入录制页后默认处于“未录制”状态；只有点击“开始录制”后才追加动作步骤
  - `Backspace / Esc / Enter` 等录制控制热键仅在“停止录制/未录制”状态下生效
- **Out-of-Scope**:
  - 暂不扩展到 `MENU_SUBMIT/MENU_CANCEL/MENU_PANE_LEFT/MENU_PANE_RIGHT`
  - 暂不新增“显示物理键名”的 UI 需求
  - 暂不改变语音关键词体系

## 1.5 Code Map (Project Topology)
- **Core Logic**:
  - `HkVoiceMod/Menu/VoiceMacroCaptureService.cs`: 录制状态机；在 `Update` 中轮询 `UnityEngine.Input.GetKeyDown(keyCode)`，捕获物理键输入。
  - `HkVoiceMod/Menu/GameKeybindNameResolver.cs`: 把物理 `KeyCode` 反查为动作标识；当前读取 `InputHandler.Instance?.inputActions` 的 `PlayerAction.Bindings`，后续应切到 `HeroActionButton + InputHandler` 原生动作入口。
  - `HkVoiceMod/Menu/VoiceSettingsMenuBuilder.cs`: 把 `VoiceMacroStep.Keys` 格式化为展示文本；展示的是动作显示名，不是物理键字符。
  - `HkVoiceMod/Input/HeroActionInputInjector.cs`: 宏执行/语音执行时真正写入 `inputActions.*`；当前仍只硬编码 `left/right/up/down/attack/jump/dash/cast`。
  - `HkVoiceMod/UI/VoiceSettingsWindowController.cs`: 录制弹窗 UI、按钮创建、弹窗开关、delay 弹窗切换与状态文案更新。
- **Entry Points**:
  - `VoiceMacroCaptureService.BeginCapture(...)`: 开始录制。
  - `VoiceMacroCaptureService.Poll() -> HandleKeyDown(KeyCode)`: 捕获本帧按下的键。
  - `GameKeybindNameResolver.TryResolveFromCurrentBindings(...)`: 用当前游戏绑定把 `KeyCode` 映射为动作枚举。
  - `VoiceSettingsMenuBuilder.FormatMacroSteps(...)`: 把步骤渲染成 `cast` / `jump` 之类的动作文本。
- **Data Models**:
  - `HkVoiceMod/Commands/HeroActionKey.cs`: 当前仅支持 8 个动作；后续应由游戏原生 `GlobalEnums.HeroActionButton` 取代。
  - `HkVoiceMod/Commands/VoiceMacroStep.cs`: 宏步骤承载动作列表；当前类型绑死 `HeroActionKey`。
- **Dependencies**:
  - `InputHandler.Instance.inputActions`: Hollow Knight / InControl 动作绑定层，代表游戏当前动作绑定。
  - `InputHandler.ActionButtonToPlayerAction(GlobalEnums.HeroActionButton)`: 游戏原生“键位页动作 -> PlayerAction”映射入口。
  - `InputHandler.GetKeyBindingForAction(PlayerAction)`: 游戏原生“动作 -> 当前键鼠绑定”查询入口。
  - `InputHandler.ActionButtonLocalizedKey(PlayerAction)`: 游戏原生动作显示/本地化辅助入口。
  - `InControl.PlayerAction.Bindings`: 动作到具体按键/设备绑定的集合。
  - `UnityEngine.Input`: 当前录制阶段用于拿“本帧按下了哪个物理键”。

## 2. Architecture (Optional - Populated in INNOVATE)
- **推荐策略：以 `GlobalEnums.HeroActionButton` 为语义真源，而不是继续维护自定义 `HeroActionKey`。**
- 原因：
  1. `HeroActionButton` 已与游戏键位页同层；本机反射确认包含 `JUMP/ATTACK/DASH/SUPER_DASH/CAST/QUICK_CAST/DREAM_NAIL/QUICK_MAP/INVENTORY/UP/DOWN/LEFT/RIGHT`。
  2. `InputHandler` 已提供 `ActionButtonToPlayerAction(...)`、`GetKeyBindingForAction(...)`、`ActionButtonLocalizedKey(...)`，可以复用游戏现有映射与显示能力。
  3. 这样“玩家改了键位 -> `PlayerAction.Bindings` 变化 -> 录制侧自动适配”天然成立，不再需要维护 `Q=Cast` 之类硬编码表。
- **录制链路目标形态**：
  1. `UnityEngine.Input.GetKeyDown(KeyCode)` 捕获本帧物理键；
  2. 枚举受支持的 `HeroActionButton`；
  3. 对每个按钮调用 `InputHandler.Instance.ActionButtonToPlayerAction(button)` 拿到 `PlayerAction`；
  4. 读取该 `PlayerAction` 当前绑定（优先 `InputHandler.GetKeyBindingForAction`，必要时回退 `Bindings` 全量比对）；
  5. 若当前物理键命中该按钮绑定，则记录 `HeroActionButton`，而不是自定义 enum。
- **回放链路目标形态**：
  - 运行时根据 `HeroActionButton` 动态拿到 `PlayerAction` 并 `CommitWithState(...)`；方向键保留 `moveVector` 特殊合成逻辑，其余按钮走统一 bool action 注入。
- **关键取舍**：
  - 录制支持列表可以按截图做 gameplay subset 过滤，但**键位映射本身**必须全部来自游戏运行时 binding，不在 mod 内维护键位表。

- **录制 UI 状态机补充**：
  1. `进入录制页`：弹窗打开，但录制服务处于 `idle`，不追加任何步骤。
  2. `点击开始录制`：进入 `recording`，按游戏当前绑定键追加动作。
  3. `点击停止录制`：回到 `idle`，此时允许 `Backspace / Esc / Enter` 作为录制界面控制热键。
  4. `点击清空`：清空当前宏步骤，但不关闭弹窗；默认保持 `idle`，避免误清空后继续采集。
  5. `打开 Delay 弹窗`：若当前在 `recording`，先进入暂停态；返回录制页后恢复到打开 Delay 前的录制/停止状态。

- **控制键规则**：
  - `recording` 状态：`Backspace / Esc / Enter` 不再被录制服务拦截，避免抢占玩家真实绑定键。
  - `idle` 状态：
    - `Backspace` = 删除最后一步
    - `Esc` = 取消本次录制并回滚到打开弹窗前的快照
    - `Enter` = 确认本次录制并关闭弹窗

## 11. Display Bugfix: UNKNOWN Fallback
- 现象：录制预览中，某些合法动作（已确认至少包含 `DOWN` / 用户按 `S`）显示为字面量 `UNKNOWN`，但动作记录与回放本身正确。
- 根因：`HkVoiceMod/Menu/GameKeybindNameResolver.cs` 中 `GetDisplayName(...)` 优先使用 `InputHandler.ActionButtonLocalizedKey(action)`；当前实现只判断“非空”，没有把 `UNKNOWN` 视为无效占位值。
- 修复策略：将 `UNKNOWN`（以及同类占位文本）视为无效显示结果，继续回退到 `action.Name`，最后再回退到本地中文默认名称。
- 影响范围：仅展示层，不改录制、注入、键位解析逻辑。

## 4. Current Findings Snapshot
- `VoiceMacroCaptureService` 录制时不是直接把 `Q` 写成 `Cast`；它先拿到 `Q`，再交给 `GameKeybindNameResolver` 去问“当前游戏绑定里，哪个动作绑定了 Q”。
- `GameKeybindNameResolver` 直接读取 `InputHandler.Instance?.inputActions`，并遍历 `PlayerAction.Bindings`。这说明它读的是当前动作绑定层，而不是 mod 自己另存的一套表。
- `VoiceSettingsMenuBuilder.FormatMacroStep(...)` 最终显示的是 `resolver.GetDisplayName(key)`，即动作显示名（如 `Cast` / `Jump`），不是按键字符（如 `Q` / `K`）。
- 当前系统只支持 `Cast`，不支持截图右侧那些额外动作；如果用户要“完全按截图动作集合录制”，现状还不够。

## 5. Verified Conclusion
- **当前读取来源**：
  - 原始键捕获：`UnityEngine.Input.GetKeyDown(KeyCode)`（物理键盘层）
  - 动作映射解析：`InputHandler.Instance.inputActions.<action>.Bindings`（HK / InControl 当前运行时动作绑定层）
- **所以 `q -> cast` 的含义**：不是“读错成了固定 Q=Cast”，而是“当前录制把你按下的 Q 反查成了它所对应的游戏动作 Cast”。
- **用户感知上的偏差点**：UI 显示的是**动作名**，不是**物理键名**。因此你看到的是 `cast`，而不是 `Q`。
- **真正的功能缺口**：
  1. `HeroActionKey` / `SupportedKeys` 只覆盖 `Left/Right/Up/Down/Attack/Jump/Dash/Cast`；
  2. 没有覆盖截图中的 `Quick Cast / Dream Nail / Super Dash / Quick Map`；
  3. 当前解析器只认 `KeyBindingSource` 且要求单键，不覆盖鼠标/手柄/复合绑定。
