- [x] 用户再次反馈已定位出新的阻塞问题：**当前三列表实现存在视觉重叠，并且输入框实际上不可编辑或极难聚焦**。后续修复优先级高于继续微调文案。

## 1. Requirements (Context)
- **Goal**: 调整自定义宏设置页，使其从“列表摘要 + 子页面编辑”改成“单页单行直接编辑”，并把新增/删除改为草稿态批量提交。
- **In-Scope**:
  - 已存在宏的 `唤醒词/指令`、`操作`、`阈值` 在主页面直接可编辑
  - 宏定义一行一个，不再进入独立宏编辑页
  - 新增宏不弹新页面/弹窗，直接在底部追加一个空白行
  - 删除宏不立即持久化，和新增/编辑一起等统一 Apply
  - 离开宏设置页时，沿用“统一处理未提交草稿”的思路，不再要求中途 Apply
- **Out-of-Scope**:
  - 不改语音识别后端编译逻辑
  - 不改宏运行时执行语义
  - 不在本轮引入新的宏步骤类型

## 1.5 Code Map (Project Topology)
- **菜单入口 / 交互组织**:
  - `HkVoiceMod/HkVoiceMod.cs`: Mod 菜单入口；`GetMenuScreen(...)` 委托给 `VoiceSettingsMenuBuilder`；`TryApplyVoiceCommandSettings(...)` 是统一提交入口，会校验设置、替换 `Settings`、重启运行时识别配置。
  - `HkVoiceMod/Menu/VoiceSettingsMenuBuilder.cs`: 当前宏页面主构建器。现状是：
    - 根页面显示 `Stop` 配置、宏列表摘要、`Add Macro`、`Apply`
    - 每个宏只展示摘要，编辑依赖 `BuildMacroEditorMenu(...)` 进入独立子页面
    - 新增/删除宏后会立刻重建根菜单
    - 退出根菜单时如果有未应用变更，会弹确认框：`Apply 并返回 / 直接返回 / 继续编辑`
  - `HkVoiceMod/Menu/VoiceMacroCaptureService.cs`: 宏录制服务；当前宏步骤追加动作依赖这里的捕获状态，UI 通过只读字段显示录制状态。
- **草稿态 / 变更检测**:
  - `HkVoiceMod/Menu/VoiceSettingsDraft.cs`: 当前页面使用的草稿容器。`FromAppliedSettings(...)` 从已应用设置克隆；`AddMacro(...)` / `RemoveMacro(...)` 只改草稿；`HasPendingChanges(...)` 比较草稿与已应用设置是否不同。
  - 这意味着“统一 Apply”基础设施已经存在，当前问题主要是 **UI 仍然拆成列表页 + 宏编辑页**，而不是数据层不支持草稿。
- **数据模型 / 校验约束**:
  - `HkVoiceMod/Commands/VoiceMacroConfig.cs`: 单条宏配置模型，字段包括 `Id / DisplayName / WakeWord / KeywordThreshold / Steps / IsPreset`。
  - `HkVoiceMod/VoiceModSettings.cs`: 宏设置总模型。关键约束：
    - `NormalizeAndValidateMacroSettings()` 当前要求每个宏必须有非空 `DisplayName`
    - `WakeWord` 不能为空且只能是中文
    - `KeywordThreshold` 必须在 `0.01 ~ 1.00`
    - `Steps` 至少一个
  - 这和“底部先插入三个空栏位、最后统一 Apply 再校验”之间存在天然张力，说明 **UI 草稿态允许空值，但最终提交前必须补齐或在 Plan 中放宽校验/补默认值策略**。
- **当前 UI 与目标差异（已定位）**:
  - 现状有单独的 `BuildMacroEditorMenu(...)` 子页面，与“所有宏都在同一页单行编辑”冲突。
  - 现状仍暴露 `DisplayName` 输入框，与“不要名字”冲突。
  - 现状一个宏会占多行：标题、阈值摘要、步骤摘要、编辑按钮、删除按钮，与“一行一个宏定义”冲突。
  - 现状新增宏会先创建默认对象 `DisplayName = "新宏"`、`KeywordThreshold = 0.1f`、空步骤，然后刷新根菜单；这与“新增时直接出现三个空栏位”不完全一致。

## 3. Detailed Design & Implementation (Populated in PLAN)
### 3.1 总体实现策略

- 菜单结构从“宏摘要列表 + 宏编辑子页”收敛为“表格化单页内联编辑”。
- 单行布局继续使用 `Satchel.BetterMenus.MenuRow`，但每个宏行**只允许三个可见列**：`命令 / 阈值 / 操作`。
- 顶部保留一行表头：`命令`、`阈值`、`操作`。
- 表格化不以“看起来像三列”为准，而以**每列控件互不覆盖、可聚焦、可编辑**为验收标准。
- `Stop` 继续固定在表格上方，独立为两项输入：`Stop 唤醒词`、`Stop 阈值`；不再保留解释性文字。
- 普通宏改为**严格三列草稿行**：
  1. `命令`：直接编辑 `WakeWord`
  2. `阈值`：直接编辑 `KeywordThreshold` 的文本草稿
  3. `操作`：单个可点击控件，点击进入录制；显示内容为当前宏步骤摘要，录制中则显示简短录制提示
- 行内不再出现 `清空步骤 / 添加延迟 / 删除 / 状态说明 / 额外标题` 等任何附加元素。
- `操作` 列的具体交互锁定为：
  - 默认显示当前录制内容摘要，空时显示 `<点击录制>`
  - 点击后开始录制，并沿用现有录制语义：按游戏当前绑定键追加步骤，`Backspace` 删除末尾，`Enter` 确认，`Esc` 取消
  - 本轮从该页面移除“显式添加延迟”的入口，以换取严格三列布局
- 新增宏时直接在列表底部追加一个空白草稿行，不弹窗，不跳页。
- 新增宏行的“命令 / 阈值 / 操作”都需要支持空白初始态：
  - `WakeWord` 为空字符串
  - `Steps` 为空列表，摘要显示 `<空>`
  - `Threshold` 通过草稿层额外维护原始文本，使 UI 可显示空字符串；真正数值字段在空白/非法输入时写入无效哨兵值，留待统一 Apply 校验失败
- 所有新增/删除/编辑都只改 `VoiceSettingsDraft`，不触发即时 Apply；统一由底部 `Apply` 或离页时的自动提交负责持久化。
- 底部按钮区改成紧凑 footer：`Add Macro`、`Delete Selected`、`Apply`、`返回`。删除按钮对“当前选中宏”生效，当前选中宏由命令/阈值编辑或操作列点击时更新。
- 如果 `MenuRow + InputField` 的组合无法稳定支持行内编辑，则允许退回到“伪表格”布局：表头一行，数据区仍按三段顺序纵向渲染，但视觉上每条宏只占一行区域，并优先保证输入可用性。

### 3.2 数据兼容策略

- `VoiceMacroConfig.DisplayName` 暂不从持久化模型删除，避免牵连运行时、旧存档与校验链路。
- 但 UI 不再暴露 `DisplayName` 输入框；提交前按以下规则补齐：
  - 若 `WakeWord` 非空，则把 `DisplayName` 自动同步为 `WakeWord`
  - 若 `WakeWord` 仍为空，则维持空值，交由统一 Apply 校验失败并阻止保存
- 这样可以满足“不要名字”的 UI 要求，同时把改动范围限制在菜单层和提交前整理阶段。

### 3.5 风险与边界

- 当前已从“内联按钮区”进一步收缩为“严格三列”，因此本轮不新增步骤 DSL 解析器，也不保留显式延迟按钮。
- 删除功能需要借助“当前选中宏”实现；这是为了同时满足“可删除宏”与“每行只有三列内容”。
- 当前最大风险已从“信息太多”转移为 `BetterMenus` 对 `MenuRow` 内嵌 `InputField` 的兼容性；若该组合天然会重叠或拦截焦点，必须换成更保守的布局实现。
- 已补充一条源码级修复：Satchel 的 `InputField` 默认 `inputBoxWidth = 300f`，对于三列表格过宽；当前实现已把命令列输入框缩到 `220f`、阈值列缩到 `120f`，并同步下调字号，以降低重叠与焦点区域互相覆盖的概率。
