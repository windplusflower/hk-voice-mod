# SDD Spec: Mod 菜单支持按操作自定义唤醒词与阈值

## 0. 🚨 Open Questions (MUST BE CLEAR BEFORE CODING)
- [x] “每种操作”的粒度采用当前 `VoiceCommand` 的 11 个固定命令：`Up / Down / Left / Right / Attack / Jump / Dash / Howl / Dive / Cast / Stop`。
- [x] “自定义唤醒词”采用**每个操作单个词条**，不做多别名。
- [x] “阈值”采用 Sherpa 关键词词条级阈值，即 `keywords_raw.txt` 每条关键词后的 `#0.05 / #0.12` 风格；不以全局 `KeywordsThreshold = 0.25f` 作为“每操作阈值”的主要表达方式。
- [x] 已确认一个关键实现前提：当前项目引用的 `sherpa-onnx.dll` 中，`KeywordResult` 公开字段只有 `Keyword`，**没有 score/confidence 输出**。因此“每操作阈值”不走 Mod 侧二次过滤，而走关键词文件生成阶段。
- [x] 用户已明确要求**游戏内自由输入文本**，因此菜单技术锁定为：`ICustomMenuMod` + `Satchel.BetterMenus`，用 `InputField` / `FloatInputField` / `MenuButton` 组合实现，而不是基础 `IMenuMod`。
- [ ] 新发现的关键阻塞：当前仓库没有“明文中文唤醒词 -> Sherpa `ppinyin` 关键词文件”的生成能力。Sherpa 官方资料显示自定义中文关键词通常需要 `text2token --tokens-type ppinyin` 之类的转换步骤；当前 C# 绑定没有直接暴露这个能力。若要支持真正的自由输入文本，需要在 Mod 内新增一套可用的关键词编译链路，或引入额外依赖来完成转换。

## 1. Requirements (Context)
- **Goal**: 让玩家在 Hollow Knight 的 Mod 菜单中，为每个操作自定义唤醒词与阈值，并在点击 `Apply` 后统一生效到运行中的语音识别后端。
- **In-Scope**:
  - 梳理当前唤醒词、阈值、运行时重载、配置持久化的真实实现位置
  - 识别菜单接入点与可行的配置 UI 方案
  - 支持在游戏内**自由输入**每个操作的单个唤醒词文本
  - 设计 `Apply` 语义：修改先停留在菜单编辑态，点击后才写入生效态并重启识别后端
  - 明确需要同步更新的内存映射与关键词文件
- **Out-of-Scope**:
  - 本轮 RESEARCH 不直接改代码
  - 本轮不更换 Sherpa 模型
  - 本轮不扩展到多语言识别或任意复杂词法生成

## 2. Architecture (Optional - Populated in INNOVATE)
- **已确认事实**:
  - 当前“唤醒词”不是配置项，而是双重硬编码：
    1. `keywords.txt` / `keywords_raw.txt` 决定模型允许识别哪些词；
    2. `SherpaKeywordSpotterBackend.CommandLookup` 决定识别出的词如何映射到 `VoiceCommand`。
  - 当前“阈值”也不是单一概念，而是两层：
    1. `keywords_raw.txt` 的词条级阈值（仓库已有实例）；
    2. `CreateKeywordSpotterConfig()` 中固定的全局 `KeywordsThreshold = 0.25f`。
  - 当前识别结果事件 `RecognizedCommandEvent` 只携带 `Command / RawText / Timestamp`，不携带 score；本地反射 `sherpa-onnx.dll` 也确认 `SherpaOnnx.KeywordResult` 公开属性只有 `Keyword`。因此“先低阈值识别，再在 Mod 内按命令二次判断阈值”的路线目前缺少数据支撑。
  - 运行时“应用设置”的主链路已经具备：`HkVoiceMod.Settings` -> `VoiceRuntimeController.ApplySettings()` -> `RestartBackend()`。
  - 仓库里尚未发现任何菜单实现代码，说明这次需求不仅是“改设置结构”，还包含“首次引入菜单 UI 层”。
  - 外部菜单方案已基本锁定：
    - 官方 `IMenuMod` 只有 `string[] Values` 式的枚举项，适合开关/预设，不适合自由文本输入。
    - `Satchel.BetterMenus` 提供 `Menu.GetMenuScreen(MenuScreen)`、`MenuButton`、`CustomSlider`、`Blueprints.FloatInputField(...)`、`StringInputFieldAttribute` 等能力，且本机游戏目录已安装 `Satchel.dll`。
- **初步架构判断**:
  - 若需求是真正的“每操作一个自定义唤醒词 + 一个自定义阈值”，最稳妥的数据模型应是：`VoiceCommand` 对应一个可序列化配置对象，而不是继续散落在 `keywords_raw.txt` 与 `CommandLookup` 两处。
  - `Apply` 不应直接绑定到每个控件的即时保存，而应采用“编辑态 -> 点击 Apply -> 校验 -> 写入 Settings -> 重建关键词资源 -> RestartBackend()` 的显式事务式流程。
  - 基于当前 Sherpa C# API 的真实暴露面，“每操作阈值”更应落在**生成 `keywords_raw.txt` / `keywords.txt` 时写入词条阈值**，而不是寄希望于运行时拿到 score 后再做二次过滤。
  - 用户已明确要求自由文本输入，因此菜单层必须支持文本编辑；基础枚举式菜单如果不支持文本输入，则不能作为最终方案。
  - 但“自由输入文本”不仅是菜单问题，还包含**关键词编译问题**：当前后端读取的是已编码的 `keywords.txt`，而不是明文中文。若没有额外编译步骤，仅把用户输入写入设置并不能让 Sherpa 正常识别新词。

## 3. Detailed Design & Implementation (Populated in PLAN)
### 3.1 Implementation Shape

本次实现采用“**已应用配置**”与“**菜单编辑态**”严格分离的结构：

- `VoiceModSettings` 只保存**已应用/已持久化**的 11 条命令配置。
- 菜单中的输入框只修改**内存 draft**，不直接改 `HkVoiceMod.Settings`。
- 点击 `Apply` 后才执行完整事务：
  1. 规范化与校验 11 条配置；
  2. 生成新的 `keywords_raw` / `keywords.txt` 运行时产物；
  3. 更新运行时“唤醒词 -> VoiceCommand”映射；
  4. 提交到 `HkVoiceMod.Settings`；
  5. 调用 `VoiceRuntimeController.ApplySettings()` 重启后端。
- 若任一步失败，则**保留旧配置与旧后端**，菜单编辑态不自动提交。

### 3.4 Validation Rules

`Apply` 前必须对 11 条配置做整体验证，失败则整批拒绝提交：

- 唤醒词规范化：
  - `Trim()`
  - 去掉所有空白字符
  - 去掉前导 `@`
  - 禁止 `#` / `@` / 换行

- 唤醒词校验：
  - 非空
  - 规范化后长度 > 0
  - 11 条之间必须唯一
  - 初版限制为当前 pinyin provider 可稳定转换的中文字符集；若包含不支持字符则拒绝 Apply

- 阈值校验：
  - 可解析为 `float`
  - 是有限数值
  - `> 0`
  - UI 与校验共同限制在 `0.01 ~ 1.00`

- 结构校验：
  - 必须刚好包含全部 11 个 `VoiceCommand`
  - 不允许缺项
  - 不允许重复 `VoiceCommand`

### 3.5 Keyword Compilation Plan

每条命令生成一行原始词条：

- `<wakeWord> #<threshold> @<wakeWord>`

然后按 Sherpa Python `text2token` 已确认算法生成 `ppinyin`：

1. 对文本执行 `pinyin(txt)` 等价步骤，得到 syllable 序列。
2. 对每个 syllable：
   - 取 `initial = to_initials(strict=false)` 等价结果
   - 取 `final = to_finals_tone(strict=false)` 等价结果
3. 按顺序输出：
   - 若 `initial` 非空，先写 `initial`
   - 再写 `final`
4. 末尾保留 `@<wakeWord>` label，供 `KeywordResult.Keyword` 回到人类可读词。

这意味着本次实现不能把 plain-text 中文直接传给当前 C# KWS API，也不能依赖虚构的 score API 在识别后再做阈值过滤。

### 3.6 Dependency Decision

关键词编译链路采用：

- **推荐方案：`IPinyinProvider` + managed provider（纯 C#，net472 兼容）**

理由：

- 不依赖 Python / `sherpa-onnx-cli` 的用户环境。
- 能和 `Apply` 流程一并在游戏内完成。
- 未来即使更换内部词典或 provider，也不会影响菜单层、设置层、后端层接口。

### 3.7 Runtime Apply Flow

1. 打开菜单时，从 `HkVoiceMod.Settings.Clone()` 创建 `VoiceSettingsDraft`。
2. 菜单输入框只更新 draft，不改 `HkVoiceMod.Settings`。
3. 点击 `Apply` 后调用 `HkVoiceMod.TryApplyVoiceCommandSettings(draftSettings)`。
4. `TryApplyVoiceCommandSettings(...)` 顺序执行：
   - 补默认值
   - 规范化与校验
   - 调用 `SherpaKeywordCompiler.Compile(...)`
   - 原子写入 `generated/keywords_raw.generated.txt` 与 `generated/keywords.generated.txt`
   - 构建新的 `wakeWord -> VoiceCommand` 映射
   - 全部成功后再写回 `HkVoiceMod.Settings`
5. 提交成功后调用 `_runtimeController.ApplySettings(Settings)`。
6. `VoiceRuntimeController.ApplySettings()` 继续复用现有“更新注入器 + RestartBackend()”链路。
7. 若失败：
   - draft 保留
   - 已应用设置不变
   - 旧 backend 不停
   - 日志明确记录失败原因

- `HkVoiceMod : Mod, IGlobalSettings<VoiceModSettings>, ICustomMenuMod`

### 3.10 Assumptions and Risks

- 初版自定义唤醒词以**中文词**为主，不扩展到任意语言/符号串。
- `VoiceCommand` 仍固定为当前 11 个操作，不增删。
- Satchel 在目标机器上已安装，且构建时可解析 `Satchel.dll`。
- 最大真实风险不是菜单，而是 **ppinyin 编译一致性**：虽然 Sherpa Python 算法已明确，但 C# provider 是否在多音字/边界字上达到足够一致，需要在实现后重点验证。
