## 2. Architecture (Optional - Populated in INNOVATE)
- **当前已确认事实**:
  - 当前问题不像简单的 spacing 微调，更像是把可编辑输入框、摘要按钮、表头列宽和菜单焦点行为硬塞进单行三列后，触发了 `BetterMenus` 的布局与交互边界。
  - 用户提出的“两行布局 + 录制弹窗”方向是合理的，因为它主动降低主页面密度，把复杂录制交互从列表行中抽走。
  - `Satchel.BetterMenus` 比公开文档展示得更强：源码层面已确认支持 `InputField`、`IntInputField`、`MenuRow`、`NavigateToMenu`、确认对话框、`StaticPanel` / 自定义 `Element` 扩展点。
  - 但它的天然强项仍然是“菜单页 + 标准控件组合”，不是“复杂实时录制弹窗系统”；目前没有现成的一等公民宏录制工作流或复合双行编辑行控件。
  - `MenuRow` 的定位更接近“单行横向组合”，不适合作为高密度可编辑表格的长期承载容器。
- **Oracle 最终建议**:
  - 最终推荐路线是 **B：混合方案**。
  - 选择理由：
    1. 纯 `BetterMenus` 虽然可以做嵌套页面、确认对话框、整数输入，但**不适合实时录制文本预览**，也没有“复合双行编辑项”这种一等控件；
    2. 完全自研 UI 会浪费当前已成型的入口、页面导航、草稿态、统一 Apply 和设置持久化链路；
    3. 因此最小可维护路径应该是：**主设置页仍留在 BetterMenus，复杂录制体验单独抽成自定义录制弹窗/面板**。
  - Oracle 给出的最小可行结构：
    - 根页面 `BuildRootMenu(...)` 保留在 BetterMenus；
    - 每个宏不再压成一条三列表格，而是改为**两个连续元素**：
      - 第一行：`命令词 + 阈值 + 录制入口`
      - 第二行：完整宏预览 / 当前录制状态
    - 点击录制后进入独立录制页或录制弹窗；
    - 录制页包含：实时预览区、`确认 / 取消 / Delay`，其中 delay 走整数输入；
    - 宏捕获继续复用 `VoiceMacroCaptureService`，但不要再尝试在主列表上每录一个键就强刷 BetterMenus 主页面。
  - Oracle 进一步指出的关键约束：
    - BetterMenus 菜单不是为高频实时刷新设计的；
    - `VoiceMacroCaptureService` 当前会监听 `Backspace / Esc / Enter`，而这些键与输入框天然冲突；
    - 因此 delay 输入时必须显式暂停录制捕获或隔离输入焦点。

- **需求变更后的最终架构判断**:
  - 继续把 BetterMenus 作为编辑器布局容器已经不成立，真实游戏内反复出现的重叠问题说明它不是这个需求的稳定承载层。
  - 新的最小可维护架构应为：
    1. `ICustomMenuMod.GetMenuScreen(...)` 保留，只负责进入 mod 配置；
    2. 真正的编辑器变成一个**自制 Unity UI 窗口**；
    3. 该窗口使用独立 `Canvas + CanvasGroup + GraphicRaycaster` 体系，不再依赖 BetterMenus 行布局；
    4. 草稿、录制、Apply、运行时设置沿用现有逻辑，避免把 UI 返工扩散成全链路重写。
  - 视觉策略采用 **v1 稳定优先**：
    - 默认使用简单自绘面板/按钮/边框，确保布局与交互先稳定；
    - 原版菜单资源复用只作为“顺手就用”的增强项，不作为阻塞条件；
    - 若低成本可复用字体，则优先保留原版风格字体；复杂装饰图、动画边框、原版按钮拼装暂不纳入 v1 blocker。

## 3. Detailed Design & Implementation (Populated in PLAN)

### 3.1 总体实现策略

- 旧的 `VoiceSettingsMenuBuilder` 不再负责真正的设置编辑界面，只保留为**启动器 / 过渡边界**，或直接被新的 launcher 类替代。
- 真正的编辑器改成一个**自制 Unity UI 窗口**，核心原则是：
  1. 布局完全由我们自己控制；
  2. 继续复用现有草稿、录制、Apply、运行时设置链路；
  3. 先把功能和稳定性做对，再考虑贴近原版视觉资源。
- v1 默认视觉方案：
  - 自绘深色半透明背景 + 主面板 + 文本标签 + 按钮 + 输入框；
  - 若能低成本复用原版字体则优先复用；
    - 原版复杂边框图片、按钮动画、原生滚动条样式都不作为首版阻塞。

- **最新入口架构判断（关键）**:
  - 如果坚持“点击 `HKVOICEMOD` 项时不进入任何独立 launcher screen，也不对底层菜单做 show/hide/nav 操作”，那么当前方案必须再变更：
    1. 放弃让 `GetMenuScreen(...)` 的返回值承担真正入口职责；
    2. 改为在菜单构建后直接 patch `HKVOICEMOD_Settings` 按钮的 `submitAction`；
    3. 点击时直接打开 overlay，自定义 UI 自己处理关闭；
    4. `GetMenuScreen(...)` 返回的 screen 退化成“理论保底对象”，正常流程下不应被进入。
  - 这条路**技术上可行，但属于 unsupported/internal patch**，不是 Modding API 官方支持扩展点。

### 3.2 建议的新 UI 结构

- **入口层**
  - HK 主菜单里的 `HKVOICEMOD` 仍然调用 `GetMenuScreen(...)`
  - 但返回的对象不再承担设置布局，只负责把流程切给自制窗口 controller

- **自制窗口层**
  - 一个常驻 `MonoBehaviour` 负责：
    - 创建/缓存 `Canvas`
    - 打开/关闭主窗口
    - 管理 modal 状态
    - 同步 `VoiceSettingsDraft`
  - 画面层级建议：
    - `Fullscreen Dimmer`
    - `Main Window Panel`
      - Header
      - Stop Config Section
      - Macro List Scroll View
      - Footer Buttons
    - `Modal Host`
      - Recording Modal
      - Delay Input Modal

- **宏行布局**
  - 每条宏是真正的两行，不再依赖 BetterMenus 行占位：
    - 第一行：`WakeWord Input + Threshold Input + 录制按钮 + 删除按钮（建议改成行内删除）`
    - 第二行：`录制摘要文本 / 当前状态`
  - 这一块用 `VerticalLayoutGroup + HorizontalLayoutGroup + LayoutElement` 构建，而不是手写 XDelta。

### 3.3 保留与替换的代码边界

- **保留**
  - `HkVoiceMod/HkVoiceMod.cs`
    - `GetMenuScreen(...)` 作为入口边界
    - `TryApplyVoiceCommandSettings(...)` 作为统一 Apply 边界
  - `HkVoiceMod/Menu/VoiceSettingsDraft.cs`
    - 整个草稿态继续复用
  - `HkVoiceMod/Menu/VoiceMacroCaptureService.cs`
    - 整个录制捕获服务继续复用
  - `HkVoiceMod/Commands/*`、`VoiceModSettings.cs`
    - 数据结构与校验继续复用

- **替换 / 新增**
  - `HkVoiceMod/Menu/VoiceSettingsMenuBuilder.cs`
    - 不再作为最终编辑器页面
    - 建议替换为 launcher-only 实现，或将其中非 BetterMenus 逻辑迁出后保留最薄入口
  - **建议新增文件**：
    - `HkVoiceMod/UI/VoiceSettingsWindowController.cs`
    - `HkVoiceMod/UI/VoiceSettingsWindowView.cs`
    - `HkVoiceMod/UI/VoiceMacroRowView.cs`
    - `HkVoiceMod/UI/VoiceSettingsLauncher.cs`（可选，若不想继续复用原 builder）

### 3.4 页面流与交互流

- **打开流程**
  - 用户点击 HK 菜单中的 `HKVOICEMOD`
  - 先进入一个官方支持的 `HKVOICEMOD` 设置页（真实 `MenuScreen`）
  - 设置页内部提供一个入口控件，用于打开/关闭自制 UI
  - 用户主动触发该入口后，再打开自制窗口，并把 `draft + mod + currentMenuScreen` 交给 controller

- **主窗口**
  - Stop 区：
    - `Stop 唤醒词`
    - `Stop 阈值`
  - 宏列表区：
    - 每条宏两行布局
    - 支持新增、删除、编辑唤醒词、编辑阈值、进入录制
  - 底部按钮：
    - `Add Macro`
    - `Apply`
    - `Discard`
    - `Close/Back`

- **录制弹窗**
  - 打开时克隆当前宏步骤快照
  - 调用 `VoiceMacroCaptureService.BeginCapture(...)`
  - 中间展示**完整步骤序列**
  - 底部按钮：`确认 / 取消 / Delay`
  - 语义：
    - `确认` = Enter，保留当前录制结果
    - `取消` = Esc，回滚到进入录制前的步骤快照
    - `Delay` = 暂停捕获，进入 delay 输入弹窗

- **Delay 输入弹窗**
  - 输入框：毫秒整数
  - 按钮：确认 / 取消
  - `确认` 后：
    - `delayMilliseconds > 0`
    - 写回 `VoiceSettingsDraft.SetPendingDelayMilliseconds(...)`
    - 追加 `VoiceMacroStep.CreateDelay(delayMilliseconds / 1000f)`
    - 返回录制弹窗并恢复捕获
    - 恢复捕获时要跳过“关闭 Delay 的同一帧按键”，避免 `Enter/Esc` 立即穿透到录制器
  - `取消`：不写入，返回录制弹窗并恢复捕获

- **关闭流程**
  - 若无未提交变更：直接隐藏自制窗口，露出当前 `HKVOICEMOD` 设置页
  - 若有未提交变更：
    - 默认仍优先沿用当前“尝试统一 Apply”逻辑
    - 若 Apply 失败，则窗口保持打开并展示错误
    - 同时必须存在一个**显式放弃更改并关闭**的路径，不能把用户困在窗口里
    - 当前选定实现：主窗口 footer 固定提供 `Discard` 按钮
  - 关闭流程不再直接处理 mod 列表或底层原始菜单
  - 关闭后自然回到当前 `HKVOICEMOD` 设置页

### 3.8 风险与验收边界

- **风险 1：输入冲突**
  - 自制窗口中的输入框与 `VoiceMacroCaptureService` 共享键盘，delay 输入期间必须暂停捕获。
- **风险 1.5：同帧热键穿透**
  - Delay 输入页关闭时，若恢复捕获过早，关闭弹窗使用的 `Enter/Esc` 会立刻触发录制确认/取消。
- **风险 2：菜单切回与生命周期**
  - 关闭窗口时必须正确返回 HK 菜单，不得留下悬空 canvas 或不可点击遮罩。
- **风险 2.1：关闭时二次切菜单**
  - 如果关闭自制 UI 时再次主动 `GoToMenuScreen(...)`，就可能造成 launcher 与原始设置页叠层。
- **风险 2.15：launcher 可见残留**
  - 即使不做菜单切换，如果 launcher screen 自己没有被隐藏，也会和底层原始设置页叠在一起。
- **风险 2.16：隐藏当前激活 screen**
  - 如果直接把当前 launcher screen `SetActive(false)`，而又不让它正常退出，就会出现当前截图里的“什么也没有”。
- **风险 2.5：launcher 隐藏范围**
  - launcher 只能隐藏自己的本地对象，不能直接操作菜单根节点的整棵层级。
- **风险 2.75：错误的打开时机**
  - 自制窗口只能由用户点击 `HKVOICEMOD` 入口显式打开，不能在游戏启动或非目标菜单状态下自动弹出。
- **风险 2.9：关闭后的残留菜单状态**
  - 关闭自制窗口后，不能留下 launcher 页与原菜单页叠层共存。
- **风险 2.95：launcher 卡住**
  - 进入 `HKVOICEMOD` 后不能停留在 launcher 文本页；launcher 只允许作为瞬时跳板存在。
- **风险 3.5：unsupported 入口 patch**
  - 直改 `HKVOICEMOD_Settings` 按钮回调虽然能更贴近用户想要的交互，但它依赖 `ModListMenu` / `MenuButton` 内部实现，未来 Modding API / Satchel 改动时可能失效。
- **风险 3.6：入口语义**
  - 如果入口被实现成“持久化设置开关”，就会引入不必要的状态同步问题；更合理的是“按钮或瞬时 toggle”，只作为 UI 打开动作。
- **风险 3：控制器导航**
  - v1 以鼠标 + 键盘稳定可用为优先；完整手柄导航不作为首版 blocker。
- **风险 4：原版美术复用成本**
  - 原版图片资源定位、拼装、适配成本可能明显高于纯功能 UI，故不作为 v1 强制项。

- **验收标准**
  - 点击 `HKVOICEMOD` 后先进入官方支持的真实设置页
  - 游戏启动时不会自动打开自制窗口
  - 不会卡在“正在打开 HKVOICEMOD 编辑器...” 页面
  - 自制窗口只能由设置页内部入口显式打开
  - 自制窗口打开/关闭时，不对 mod 列表页做显式 show/hide/nav 操作
  - Stop 区可编辑
  - 宏列表能完整显示所有已有宏，而不是只露出一条
  - 普通宏列表稳定显示为两行布局，不再出现重叠
  - 录制弹窗能展示完整步骤序列，且说明文案、文本区、按钮区互不遮挡
  - `确认 / 取消 / Delay` 与 Enter / Esc 语义一致
  - Delay 输入框能正常回显输入内容
  - 每条宏中唤醒词输入框宽度足够，不再过窄
  - delay 以毫秒整数输入并正确写入步骤
  - Delay 弹窗关闭时不会误触发录制确认/取消
  - 即使 Apply 失败，用户仍能明确地放弃未保存更改并退出
  - 关闭自制窗口后不会出现菜单页叠层
  - 点击 Back/Discard 关闭时，只隐藏自制 UI，不再额外触发菜单跳转
  - 点击 Back/Discard 关闭后，不会看到 launcher 文本页重新浮出来，也不会只剩空白背景
  - 关闭自制窗口后会自然回到 `HKVOICEMOD` 设置页，而不是 launcher 页面或空白背景
  - 再次进入自制窗口不会报错
  - `Apply` 后设置真正写回并作用于运行时
