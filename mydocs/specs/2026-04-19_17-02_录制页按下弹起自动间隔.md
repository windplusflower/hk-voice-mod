# SDD Spec: 录制页按下弹起自动间隔

## 0. 🚨 Open Questions (MUST BE CLEAR BEFORE CODING)
- None
- 已确认目标：录制侧不再只记录“动作步骤 + 手动插入延迟”，而是改为记录 `按下 / 松开` 事件以及两事件之间的真实时间间隔。
- 已确认边界：忽略首个按键事件之前的空闲停顿，也忽略最后一个按键事件之后的空闲停顿。
- 已确认编辑语义：用户编辑时不需要区分按下和松开；编辑该对事件中的任一端，都会把整对事件的动作键同步替换为新键。
- 已确认破坏性策略：不做旧宏兼容；旧默认配置直接改写为新事件流序列，本地已存在的旧配置按“清空并重置到新 schema”处理。

## 1. Requirements (Context)
- **Goal**: 把录制页宏录制能力升级成类似按键精灵的事件流录制：记录按下、松开与事件间隔，同时保留现有逐项编辑能力，并把按下/松开视为同一逻辑键对进行联动编辑。
- **In-Scope**:
  - 改造录制捕获逻辑，从仅监听 `GetKeyDown` 扩展到 `GetKeyDown + GetKeyUp + 时间戳/间隔`。
  - 改造宏步骤数据模型，使其能表达事件型动作与事件间延迟，而不再只依赖 `KeyPressMode + DurationSeconds` 推导。
  - 改造录制弹窗步骤列表，使动作步骤在展示与编辑时能映射到“逻辑键对”。
  - 改造宏回放逻辑，使其按事件时间线驱动输入注入，而不是把动作步骤直接翻译成一次 `Tap/TimedHold/ContinuousHold`。
  - 保留 Delay/时序可编辑性，但默认来源改为录制自动生成的间隔。
  - 废弃旧宏兼容路径：旧默认配置改写为新事件流默认序列，本地旧配置在加载时清空并重置为新版本默认配置。
- **Out-of-Scope**:
  - 不修改语音唤醒词、阈值、宏列表外层布局。
  - 不改变 `InputHandler` / `HeroActionInputInjector` 以外的游戏输入注入基础设施。
  - 不在本轮引入鼠标、手柄、多设备组合录制；仍聚焦当前已支持的游戏键位动作集合。
  - 不保留旧 `Steps` 宏的运行时 fallback，也不做旧 `ContinuousHold` 语义兼容迁移。

## 2. Architecture (Optional - Populated in INNOVATE)
- **当前已确认事实**:
  1. 录制捕获层只监听 `GetKeyDown`，因此现状根本拿不到松开事件，也无法自动计算真实按住时长。
  2. 当前 UI 编辑语义是一条 `Action` 步骤对应一个动作键；这和“一个逻辑键要展开为 press/release 两条事件”存在一对多映射冲突。
  3. 当前回放层把宏视为“动作 + delay”的排程，输入注入层也只接受 `press mode` 语义；如果要精确回放按下/弹起，运行时链路必须同步改成事件模型。
  4. 现有 `Delay` 步骤已经可编辑，但它表达的是显式独立步骤，不是“两个输入事件之间的时间差”这一底层事实。
- **复杂度判断**:
  - 这是一个 **复杂、多模块、跨模型** 的改造，至少涉及 `Capture -> UI Editor -> Data Model -> Runner -> Injector -> Validation` 六段链路，不适合直接进入 EXECUTE。
- **方案对比**:
  - **方向 A：底层事件流真源（推荐）**
    - 宏内部真实存储 `KeyDown/KeyUp + 相对时间/事件间隔`；UI 录制页再把可配对事件折叠显示成逻辑键对。
    - **优点**:
      - 完全符合用户表述：录制的是“按下/弹起”，不是推导后的 pressMode。
      - “忽略首前尾后停顿”实现简单：录制结束后对首个有效事件与最后一个有效事件做 trim 即可。
      - 编辑时只要维护 `pairId` 或 `pressId`，就能天然实现“改任一端，同步改整对”的语义。
      - 未来若要支持重叠按键、交错按住、精细时序，不需要再推翻模型。
    - **代价/风险**:
      - `VoiceMacroStep`、`VoiceMacroRunner`、`HeroActionInputInjector`、`VoiceModSettings.ValidateMacroStep(...)` 都要一起改。
      - 录制页 UI 需要从“按 step 编辑”升级成“按逻辑键对展示，但底层回写事件流”。
  - **方向 B：逻辑键对真源**
    - 宏内部存储“动作键 + 按住时长 + 与下一逻辑动作的间隔”；录制阶段仍观察 down/up，但落盘时压缩成一对结构。
    - **优点**:
      - UI 与编辑逻辑更直观，数据结构更接近用户心智模型。
      - 对现有 `Delay + Action` 风格更像增量演进，短期实现压力略低。
    - **代价/风险**:
      - 本质仍是“事件录制后再次压缩回推导模型”，会丢掉真实事件流细节。
      - 若后续遇到按键重叠或异常配对，模型会迅速变得别扭。
      - 回放层最终仍要做额外推导，复杂度只是被推迟，而不是消失。

## 2.1 Recommended Strategy
- **推荐选择：方向 A（底层事件流真源）**
- **原因**:
  1. 用户需求已经明确把“录制按下与弹起”提升为第一语义，不再是现有 `PressMode/DurationSeconds` 的隐式推导；继续用压缩模型会在语义上绕一圈再还原，不值得。
  2. 当前仓库已经有较完整的录制页逐行编辑壳与 `BeginSingleKeyCapture(...)` 机制，UI 层可以承接“逻辑键对折叠编辑”；真正缺的是底层数据真源。
  3. 现有回放与校验链路虽然需要改，但都集中在宏专用链路，不会波及语音命令旧链路（`VoiceCommandMap/KeyActionProfile` 仍可保持兼容）。

## 2.2 Recommended Shape
- **底层存储**：事件流真源
  - 每个动作事件至少包含：`ActionButton`、`EventPhase(Down/Up)`、`DelayBeforeSeconds` 或 `RelativeTimeSeconds`、`PairId`。
  - 其中 `PairId` 用于把一次逻辑按压的 down/up 关联起来，服务编辑器联动。
- **录制后处理**:
  - 录制时捕获所有有效 down/up 事件与相邻时间差。
  - 停止录制时执行 normalize：
    - 去掉首个有效事件之前的空闲；
    - 去掉最后一个有效事件之后的空闲；
    - 过滤 auto-repeat；
    - 补齐/校验配对关系。
- **编辑器投影层**:
  - 录制页列表默认显示“逻辑键对行”，每行展示：键名、按下到松开的持续时长、到下一逻辑事件的间隔。
  - 点击该行改键时，不区分 down/up；而是按 `PairId` 批量更新同一对事件的 `ActionButton`。
  - 若用户点击的其实是该对中的 release 端，效果与点击 press 端完全一致。
- **运行时回放**:
  - `VoiceMacroRunner` 改为按事件时间线调度。
  - `HeroActionInputInjector` 新增明确 API：`PressActionButton(...)` / `ReleaseActionButton(...)`，不再完全依赖 `PressMode/DurationSeconds` 推导。

## 2.3 Breaking-Change Boundary
- **兼容策略已取消，改为 schema 切换**:
  - 新版本只接受事件流宏（`KeyEvents`）作为有效宏表示。
  - 旧 `Steps` 仅允许在代码删改过渡期短暂保留字段定义，不再作为运行时 fallback。
  - 加载到旧 schema 或检测到 legacy 宏时，直接清空旧宏配置并切到新默认事件流配置。
- **本地旧配置清理策略**:
  - 在 `VoiceModSettings` 中新增显式 schema 版本字段，例如 `MacroStorageVersion`。
  - 在 `HkVoiceMod.OnLoadGlobal(...)` / `PrepareSettingsForRuntimeCore(...)` 早期检查：
    - 若 `MacroStorageVersion < CurrentEventStreamVersion`；或
    - 任一宏仍含 legacy `Steps`；或
    - 仍依赖 `CommandKeywordConfigs -> MigrateLegacyCommandConfigsIfNeeded()` 路径，
    - 则直接重建为新版本默认配置，并覆盖当前内存中的旧配置。
  - 这里的“清空本地旧配置”定义为：本次加载后旧配置不再进入运行时，后续正常保存时只会写出新 schema。
- **新的默认配置来源**:
  - 原有默认宏语义保留，但落盘形式一律改为 `KeyEvents` 序列。
  - 例如原先 `攻击` / `冲刺` / `法术` / `跳跃` 等默认宏，不再通过 `CreateActionStep(...)` 生成 `Steps`，而是生成对应的 `Down -> delay -> Up` 事件流序列。
- **非目标边界**:
  - 语音命令即时触发链路可以继续使用旧 `KeyActionProfile/PressMode`，本轮只改“宏录制/宏回放/宏持久化”链路。

## RESEARCH Findings Snapshot
- 当前仓库已经在 `2026-04-19_13-14_录制页宏序列列表可编辑化.md` 中完成“逐行编辑动作/延迟”的 UI 能力，这次改造必须复用该交互壳，而不是重做录制页。
- 现有录制页点击动作行时，会调用 `VoiceMacroCaptureService.BeginSingleKeyCapture(...)`，并在 `ApplyActionStepEdit(...)` 中用 `CreateActionStep(...)` 直接替换整条 `Action` 步骤；这正好是后续“按下/松开成对同步修改”需要改造的落点。
- 现有宏执行链路中，`VoiceMacroRunner.QueueMacro(...)` 会把 Delay 叠加进时间轴，再把 Action 在单点时间触发；因此自动记录间隔若继续沿用现模型，会天然被拆成“Delay + Action + Delay + Action”。
- `HeroActionInputInjector` 当前并没有“显式按下某键 / 显式松开某键”的公开接口，只有“给出 press mode 后推导状态”；若选择事件流真源，这里需要新增明确的 press/release 注入 API。
- `VoiceModSettings.ValidateMacroStep(...)` 目前禁止 `ReleaseContinuous` 出现在宏步骤里，也要求 `Tap/TimedHold` 具备合法持续时间；这说明现校验完全服务于旧模型。
- `VoiceSettingsDraft.StepChanged(...)` 当前只比较 legacy `Keys`，没有比较当前实际使用的 `ActionButtons`；如果不顺手修正，后续成对编辑即使生效，也可能不被识别为 pending changes。
- 当前编辑 UI 已经具备“动作改单键等待态”与“延迟逐项编辑弹窗”两套基础设施，因此这次不需要重建录制页，只需替换其底层数据投影与保存逻辑。

## 2.5 External Reference Snapshot
- **事件采集建议**:
  - 应用内键盘录制优先依赖标准 `keydown / keyup`，建模主键优先使用 `KeyboardEvent.code`，`key` 仅作显示补充。
  - 如果未来需要“应用失焦也能录”的全局键盘录制，Web 环境本身做不到，需 native hook（如 Electron + `uiohook-napi/iohook`）。
- **时间模型建议**:
  - 用单调时钟 `performance.now()` 记录相对时间更稳，`event.timeStamp` 可保留为调试字段，但不建议单独作为回放真源。
  - “忽略首前尾后空闲”更适合在录制完成后做 trim/normalize，而不是采集时提前裁剪。
- **抗重复输入建议**:
  - 必须过滤 auto-repeat：`keydown` 若 `repeat == true` 直接忽略，或维护 `pressedSet` 保证一次按住只记录一个 down。
- **编辑模型建议**:
  - 推荐分两层：底层保留 raw event（down/up + tRel），编辑层折叠成 `LogicalPress`（一对事件 + 持续时间），这样最贴近你要的“编辑任一端都会联动整对键值”。
- **参考来源**:
  - MDN KeyboardEvent: https://developer.mozilla.org/en-US/docs/Web/API/KeyboardEvent
  - MDN performance.now: https://developer.mozilla.org/en-US/docs/Web/API/Performance/now
  - MDN Event.timeStamp: https://developer.mozilla.org/en-US/docs/Web/API/Event/timeStamp
  - Electron keyboard shortcut / before-input-event: https://www.electronjs.org/docs/latest/tutorial/keyboard-shortcuts
  - uiohook-napi: https://github.com/snosme/uiohook-napi
  - iohook: https://wilix-team.github.io/iohook/

### 3.3 Breaking-Change Strategy
- **总原则**：新版本只认 `KeyEvents`，不再接受旧 `Steps` 作为有效宏表示。
- **启动行为**:
  - 读到旧 schema / legacy 宏 / 旧命令配置迁移路径时，直接 reset 到新默认事件流配置。
- **默认配置改写**:
  - 原有默认宏语义保留，但存储形式一律改为 `KeyEvents` 序列。
- **明确不做**:
  - 不做旧宏自动迁移。
  - 不保留旧 `Steps` 运行时 fallback。
  - 不保证本地旧宏内容延续到新版本。
  - 不改 `VoiceCommandMap/KeyActionProfile` 的旧语音命令即时触发链路。

### 3.4 Interaction Contract
- **录制中**:
  - 记录所有有效 `Down/Up` 事件。
  - 第一个有效事件前无 delay；最后一个有效事件后无 delay。
  - auto-repeat 不得生成额外 `Down` 事件。
- **停止录制后编辑**:
  - 点击 `ActionEvent` 行进入改单键模式；收到新动作键后，同一 `PairId` 的 `Down/Up` 两条事件一起改。
  - 点击 `Delay` 行只修改该事件前置间隔。
  - 编辑时不要求用户区分按下还是松开。
- **取消录制**:
  - 回滚到 `_recordingStartSnapshot`。
- **Stop 指令运行时行为**:
  - 清空未执行事件，并释放所有当前由事件流宏持有的按键状态。

### 3.5 Risks & Controls
- **风险 1：同帧 Down/Up 被吞**
  - 控制：`VoiceMacroRunner.Tick(...)` 对同帧到期的 `Down/Up` 需要保证至少一帧按下可见态。
- **风险 2：KeyUp 重新解析导致配对漂移**
  - 控制：`VoiceMacroCaptureService` 在 `KeyDown` 时缓存 `ActionButton + PairId`，`KeyUp` 只回查缓存，不重跑绑定解析。
- **风险 3：变更检测漏判**
  - 控制：`VoiceSettingsDraft` 同时比较 `ActionButtons` 与 `KeyEvents`。
- **风险 4：旧 continuous hold 语义被误迁移**
  - 控制：本轮直接放弃兼容，不做迁移；检测到旧配置后 reset 到新默认事件流配置。
