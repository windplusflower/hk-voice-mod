# SDD Spec: 录制页宏序列列表可编辑化

## 0. 🚨 Open Questions (MUST BE CLEAR BEFORE CODING)
- None
- 已确认边界：**只修改录制弹窗内**的宏序列显示与编辑方式；主窗口外层宏列表继续保留现有摘要文本，仍由 `VoiceSettingsMenuBuilder.FormatMacroSteps(...)` 输出括号分隔格式。
- 已确认交互：仅在**非录制状态**允许逐行手动编辑；点击动作键后进入“等待新键输入”状态，点击延迟后进入数值修改弹窗。

## 1. Requirements (Context)
- **Goal**: 把录制弹窗中当前的整段字符串预览，改成“一个步骤一行”的可读、可编辑列表；在停止录制/未录制状态下支持逐项修改动作键或延迟毫秒数。
- **In-Scope**:
  - 改造 `VoiceSettingsWindowController` 内录制弹窗的宏序列显示结构
  - 为动作步骤提供逐行可点击编辑入口
  - 为延迟步骤提供逐行可点击编辑入口，并复用/扩展现有 Delay 弹窗
  - 保持 `VoiceMacroStep`、`VoiceMacroConfig`、`VoiceSettingsDraft` 的数据一致性
  - 保持外层宏列表摘要仍使用括号包裹、逗号拼接的原格式
- **Out-of-Scope**:
  - 不修改主窗口外层宏列表行布局
  - 不修改 `VoiceSettingsMenuBuilder.FormatMacroSteps(...)` 的摘要展示协议
  - 不改录制服务的核心捕获规则，除非为“非录制状态下逐项编辑”补充最小必要接口
  - 不改运行时宏回放逻辑

## 1.5 Code Map (Project Topology)
- **录制页 UI 主控（本轮核心）**:
  - `HkVoiceMod/UI/VoiceSettingsWindowController.cs`
    - `BuildRecordingModal(...)`：当前录制弹窗结构；中间只有一个 `RecordingPreviewText` 文本区域，没有逐项行控件。
    - `OpenRecordingModal(...)`：打开录制弹窗时创建 capture session，并保存 `_recordingStartSnapshot`。
    - `RefreshDynamicContent()`：当前用 `VoiceSettingsMenuBuilder.FormatMacroSteps(...)` 把全部步骤拼成一段字符串塞进 `_recordingPreviewText`。
    - `OpenDelayModal()` / `ConfirmDelayInput()` / `CloseDelayModal(...)`：现有 delay 编辑/插入弹窗流程，已具备整数输入与暂停录制能力。
    - `BuildMacroRow(...)` / `BuildMacroSummaryText(...)`：主窗口外层宏列表显示逻辑；这是**必须保持不变**的外围摘要层。
- **展示格式边界（必须保持对外不变）**:
  - `HkVoiceMod/Menu/VoiceSettingsMenuBuilder.cs`
    - `FormatMacroSteps(...)`：把宏步骤格式化成 `(<step>),(<step>)` 风格字符串。
    - `FormatMacroStep(...)`：动作步骤与 delay 步骤的单项字符串格式化。
    - 该文件同时服务主窗口宏摘要与录制弹窗当前预览；本轮用户已明确要求仅改录制弹窗显示方式，因此不应直接改这里。
- **数据草稿层 / 步骤替换边界**:
  - `HkVoiceMod/Menu/VoiceSettingsDraft.cs`
    - `CloneMacroSteps(...)`：录制弹窗打开时用于保存回滚快照。
    - `ReplaceMacroSteps(...)`：已提供整组步骤替换能力，可作为逐项编辑后统一回写的安全边界。
    - `SetPendingDelayMilliseconds(...)`：delay 编辑时已有的毫秒草稿接口。
- **宏模型**:
  - `HkVoiceMod/Commands/VoiceMacroConfig.cs`：每条宏持有 `Steps`。
  - `HkVoiceMod/Commands/VoiceMacroStep.cs`：单步模型，`StepKind` 区分 `Action/Delay`；动作使用 `ActionButtons`，延迟使用 `DelaySeconds`。
- **录制捕获服务（行为约束）**:
  - `HkVoiceMod/Menu/VoiceMacroCaptureService.cs`
    - `IsCapturing(...)` / `HasCaptureSession(...)` / `IsCaptureSuspended(...)`：可用于判定当前是否允许手动编辑。
    - `StartCapture(...)` / `StopActiveCapture(...)` / `SuspendCapture()` / `ResumeCapture()`：delay 弹窗已使用 suspend/resume；若动作编辑需要“等待下一次键输入”，应优先在现有 capture 语义上做最小扩展，而不是新造第二套输入监听。
    - `HandleKeyDown(...)`：当前在 `_isCaptureActive == false` 时只处理 `Backspace/Esc/Enter` 控制键，不会把普通按键映射成动作，因此动作手动改键目前**没有现成入口**。
- **动作键显示与解析**:
  - `HkVoiceMod/Menu/GameKeybindNameResolver.cs`
    - `GetDisplayName(HeroActionButton)`：动作步骤 UI 行应继续显示这个动作名，而不是物理键字符。
    - `TryResolveFromCurrentBindings(KeyCode, out HeroActionButton, out string)`：现有“物理键 -> 当前游戏绑定动作”的解析入口，可复用于手动改键。

## 2. Architecture (Optional - Populated in INNOVATE)
- **当前已确认事实**:
  1. 录制弹窗当前不是列表，而是 `RecordingPreviewText` 单一富文本块；所以用户无法定位单个步骤，更无法逐项编辑。
  2. 外层宏列表摘要文本由 `BuildMacroSummaryText(...) -> VoiceSettingsMenuBuilder.FormatMacroSteps(...)` 驱动；如果直接改 `FormatMacroSteps(...)`，会把主窗口外层摘要一并改掉，违反用户边界。
  3. delay 编辑已有成熟弹窗与 suspend/resume 流程，可复用，不需要重新造第二个数值编辑系统。
  4. 动作手动改键还没有现成 UI/状态机支持；现有 capture service 只能在“active recording”时把普通按键解析成动作，因此手动改键大概率需要补一个“单次等待下一键”的轻量模式。
- **推荐方向（研究结论）**:
  - 在 `VoiceSettingsWindowController` 内把录制预览区从“单文本”升级为“滚动列表容器 + 每步一行 widget”。
  - 每一行根据 `VoiceMacroStep.StepKind` 分成两类：
    - `Action`：显示序号、类型标签、当前动作名按钮；在未录制状态下点击后进入“等待输入新键”态。
    - `Delay`：显示序号、类型标签、当前毫秒值按钮；点击后打开现有 Delay 弹窗，但改为“编辑当前步骤”而不是“追加新步骤”。
  - 主窗口外层摘要仍继续调用 `FormatMacroSteps(...)`，录制弹窗则单独走新列表渲染逻辑，彻底隔离两层显示。

## RESEARCH Findings Snapshot
- `BuildRecordingModal(...)` 当前只创建了 `RecordingPreviewText` 文本，没有任何步骤级按钮或行容器，因此“点某个键改键 / 点延迟改数值”在现状中无处挂接。
- `RefreshDynamicContent()` 每帧直接把 `_recordingMacro.Steps` 整体格式化为字符串；这说明录制弹窗当前是**纯展示**，不是结构化编辑界面。
- `VoiceSettingsDraft.ReplaceMacroSteps(...)` 与 `_recordingStartSnapshot` 已经提供了“编辑期可回滚”的基础设施，因此逐项编辑不需要改持久化模型。
- `VoiceMacroCaptureService.HandleKeyDown(...)` 在非录制态只响应 `Backspace/Esc/Enter`，这是手动改键的唯一明显缺口。
- `OpenDelayModal()` 当前默认绑定 `_delayMacro = _recordingMacro`，并在确认时通过 `AppendDelayStep(...)` 追加新 delay；若要支持编辑已有 delay，需要把弹窗上下文从“宏级追加”扩展成“宏级追加 / 步骤级修改”两种模式。

## 3. Detailed Design & Implementation (Populated in PLAN)

### 3.1 File Changes
- `File: HkVoiceMod/UI/VoiceSettingsWindowController.cs`
  - **字段新增/替换**:
    - 把录制弹窗的单文本预览引用扩展为“步骤列表容器 + 行缓存 + 编辑上下文”。
    - 新增动作编辑上下文：当前等待改键的步骤索引。
    - 新增 delay 编辑上下文：当前是否在编辑已有 delay，以及目标步骤索引。

      - 把 `RecordingPreviewText` 文本预览改为滚动列表容器。

      - 统一在状态切换时清掉待处理的逐项编辑状态，避免“开始录制时仍处于等待新键输入”。
    - `OpenDelayModal()` / `ConfirmDelayInput()` / `CloseDelayModal(...)`
      - 保留“插入 delay”路径；新增“编辑已有 delay”路径，按上下文决定是 append 还是覆盖现有步骤。

    - `HandleKeyDown(KeyCode keyCode)`：优先处理单次按键等待模式；`Escape` 取消，其余普通键若能解析到 `HeroActionButton` 就回调并退出等待。

### 3.2 Interaction Contract
- **录制中**:
  - 逐行列表仍可见，但所有步骤编辑按钮禁用。
  - 顶部说明继续强调“先停止录制再编辑”。
- **未录制/停止录制**:
  - 点击动作步骤 -> 进入“等待新键输入”状态；该行按钮文字切为“按下新键…”，`Esc` 取消。
  - 点击 delay 步骤 -> 打开 Delay 弹窗，标题改为“编辑延迟”，确认后直接覆盖该步骤的 `DelaySeconds`。
  - 点击“插入延迟”按钮 -> 保持现有 append 语义，标题为“添加延迟”。
- **取消/关闭**:
  - 关闭录制弹窗或切换回录制状态时，任何待处理的动作改单键状态都必须清掉。
  - `取消录制` 仍继续依赖 `_recordingStartSnapshot` 回滚整个步骤列表。

### 3.3 Non-Goals / Risk Controls
- 不修改 `VoiceSettingsMenuBuilder.FormatMacroSteps(...)`，确保主窗口外层宏摘要继续保持括号分隔格式。
- 不新增新的持久化字段；步骤修改直接落在当前 `_recordingMacro.Steps` 草稿对象中。
- delay 编辑只改 `Delay` 步骤的毫秒值，不借机重排步骤。
- 动作编辑采用“替换成新的默认动作步骤”策略：即沿用 `VoiceSettingsMenuBuilder.CreateActionStep(...)` 的默认 press mode / duration 规则，而不是盲保留旧动作的按住时长语义。
